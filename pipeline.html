Pipeline Learnings

This site builds itself.

How this site stays in sync

This portfolio is generated from a single source-of-truth data file (data/resume.json) using a hub-and-spoke pipeline. One edit propagates to every surface — the website, my CV (PDF & DOCX), a knowledge graph, and LinkedIn-ready text — automatically on every push via GitHub Actions.

Principle: DRY — every piece of knowledge has a single, unambiguous, authoritative representation (Hunt & Thomas, The Pragmatic Programmer, 1999).
data/resume.json ← SINGLE SOURCE OF TRUTH (edit only this) | +----------+-----------+------------+-----------------+ v v v v v 7 HTML pages CV .tex→PDF CV .docx linkedin.md knowledge-graph/ (the site) (LaTeX/CI) (python-docx) (paste blocks) cv-graph.json On every push → GitHub Actions rebuilds all spokes & deploys the site.

The spokes

7 HTML pages

Rendered from Jinja2 templates with schema.org/JSON-LD structured data for SEO.

CV → PDF

LaTeX generated from resume.json and compiled by TeX Live in CI.

CV → DOCX

python-docx rendering, mirroring the PDF layout.

Knowledge Graph

A property graph (schema.org + JSON Resume + ESCO ontology) regenerated from the same data.

LinkedIn blocks

Paste-ready, keyword-optimized headline/about/experience text (LinkedIn has no public write API, so this stays assisted-manual).

Why it matters

This portfolio is itself a demonstration of the work I do: turning a manual, error-prone, duplicated process (updating a résumé in five places) into an automated, single-source-of-truth system with continuous deployment — the same pattern behind the operational-quality systems I build at Amazon.
